# Mavric Templates

Templates used by the Mavric toolchain to generate consistent project artifacts.

## Available Templates

| Template | Purpose | Generated By |
|----------|---------|--------------|
| `constitution.md` | Project principles and rules | `/init` |
| `discovery-template.md` | Requirements document structure | `discovery-interviewer` |
| `technical-plan.md` | Architecture and planning | `/plan` |
| `roadmap-template.md` | Phased delivery plan | `saas-project-orchestrator` |
| `roadmap-screens-first.md` | Screens-first 6-phase approach | `saas-project-orchestrator` |
| `ralph-saas-prompt.md` | AI agent prompt for SaaS builds | Manual |
| `scenario-template.feature` | Gherkin test structure | `test-generator` |
| `quickstart.md` | Critical path validation | `saas-project-orchestrator` |
| `verification-report.md` | Artifact consistency report | `/verify` |

### Screens-First Development Templates

The **screens-first** approach builds all UI before backend:

1. **`roadmap-screens-first.md`** - 6-phase roadmap:
   - Phase 1: Foundation & Layout (no auth)
   - Phase 2: All Screens (mock data)
   - Phase 3: Backend Schema & API
   - Phase 4: Billing & Payments
   - Phase 5: Authentication
   - Phase 6: Polish & QA

2. **`ralph-saas-prompt.md`** - Autonomous AI agent prompt template with:
   - Project context placeholders
   - Tech stack (non-negotiable)
   - Phase-by-phase deliverables
   - Domain knowledge sections
   - Working conventions

## Customizing Templates

### Project-Level Customization

Copy templates to your project's `.claude/templates/` directory to customize:

```bash
# Copy a template to your project
cp devenv/.claude/templates/constitution.md my-project/.claude/templates/

# Edit to match your project's needs
# The toolchain will use your customized version
```

### Template Variables

Templates use placeholder variables in `[BRACKETS]`:

| Variable | Description |
|----------|-------------|
| `[PROJECT_NAME]` | Name of your project |
| `[DATE]` | Current date |
| `[USER_NAME]` | User/stakeholder name |
| `[ENTITY]` | Entity name (dynamic) |
| `[WORKFLOW]` | Workflow name (dynamic) |

### Template Sections

Most templates follow this structure:

1. **Header** - Title, metadata, status
2. **Summary** - Executive overview
3. **Main Content** - Detailed sections
4. **Appendix** - References, history
5. **Sign-off** - Approval section

## Template Usage

### During `/init`

Creates project with:
- `.claude/constitution.md` (from template)
- Directory structure
- `.gitignore`
- `README.md`

### During Discovery

`discovery-interviewer` fills:
- `docs/discovery/discovery-notes.md`
- `docs/discovery/discovery-summary.md`

### During Test Generation

`test-generator` creates:
- `docs/scenarios/api/*.feature`
- `docs/scenarios/ui/*.feature`
- `docs/scenarios/e2e/*.feature`

### During Planning

`/plan` command creates:
- `docs/plans/technical-plan.md`
- `docs/plans/roadmap.md`

### During Verification

`/verify` command creates:
- `docs/verification-report.md`

## Best Practices

### 1. Don't Modify autogen Templates

The templates in `devenv/.claude/templates/` are source templates. Copy them to your project before modifying.

### 2. Keep Templates Updated

When updating project structure, update templates to match.

### 3. Use Consistent Naming

Follow the established naming conventions in templates.

### 4. Include Sign-off Sections

Templates with approvals help track decisions.

## Template Development

### Adding New Templates

1. Create template in `devenv/.claude/templates/`
2. Use `[VARIABLE]` placeholders for dynamic content
3. Add to this README
4. Update relevant skill to use template
